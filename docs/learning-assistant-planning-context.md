# 学习助手添加/立项功能：用户背景摘要

来源文件：

`/Users/cpt/Library/Mobile Documents/iCloud~md~obsidian/Documents/CMU/26赛季bw/proj - 学习助手.md`

用途：

这份摘要是给后续 OpenSpec / Codex 设计迭代使用的背景材料。它不是需求文档，也不是完整 backlog。它的作用是把原始 Obsidian 笔记里和“学习助手添加/立项功能”相关的真实目标、学习对象、资料类型和约束先提炼出来，避免每次 prompt 都塞一大段原始文件分析任务。

## 1. 当前最重要目标

用户到毕业前约两个月，最重要主线是为七月中开始的互联网秋招提前批做准备，目标岗位是 agent 开发。

这一目标需要被拆成可持续推进的周计划和日计划，而不是停留在“我应该准备秋招”这种抽象目标上。

核心准备方向包括：

- 力扣
- agent 项目学习
- 改造已有项目
- agent 相关八股
- 后端八股
- 简历项目打磨

最紧急痛点不是判断“这些事值不值得做”，而是把已经大致确定要做的事拆成阶段、里程碑、每日任务，并能按设定期限推进。

## 2. 用户真正想解决的规划问题

用户不想每天把大量脑力花在“今天到底干什么”“这个项目怎么拆”“怎么在期限内做完”上。

设计时应优先解决：

- 输入一个学习/项目目标后，如何生成期限驱动的执行计划
- 如何从 deadline 倒推出阶段、里程碑和每日任务
- 如何根据每天可用时间安排任务量
- 如何区分必须执行的任务、参考资料、以后再看的资源
- 如何在进度落后、状态很差、任务变多时调整计划
- 如何设计低能量保底任务和减载机制，避免再次因为透支而长期停摆
- 如何让计划既有约束力，又不变成另一个吵闹的维护负担

设计时不应把重心放在哲学层面的价值判断，例如“这个东西到底值不值得学”或“它和人生目标有什么关系”。这些可以作为未来优化事项，但第一版不应让 AI 自动替用户决定。

## 3. 重要学习/项目对象

### 3.1 Agent 开发与项目能力

这是当前最重要的学习方向，和秋招目标直接相关。

用户提到的可能学习路径和资料包括：

- 跟一个教程 repo 从头到尾做，例如 `adongwanai/AgentGuide`
- `easyagent`，从零开始搭建 Claude Code，学习企业级 agent 搭建能力
- `learn claude code`
- 研究 `superpowers`、`gsd`、`gstack`、`openspec` 的区别、取舍和组合方式
- 研究 `forrestchang andrej-karpathy-skills`
- 啃一个流行 agent 框架的原理，例如 superpowers，并尝试为了自己的项目需求魔改或优化
- 学更底层的开源 agent 运行时框架，例如 FairyClaw，甚至尝试贡献代码
- 学 agent 记忆与 context 管理，用于自己的项目，也可用于面试表达
- 研究 spec-driven 相关资料
- 学会辨别什么是好 skill，以及什么时候需要 skill

相关资料类型包括 GitHub repo、文章、视频、HTML 资料、开源框架、已有项目源码。

### 3.2 力扣

面向国内面试，用户倾向于：

- Hot 100 反复刷
- 灵茶山基础精炼反复练

对学习助手而言，力扣更像是稳定训练计划，而不是资料管理问题。需要支持每日/每周节奏、复习和进度记录。

### 3.3 Agent 和后端八股

用户想准备 agent 和后端八股。

其中一个想法是做一个 agent 去爬小红书面经。这个方向既可能是面试资料收集工具，也可能是一个可展示的 agent 项目。

设计时应注意：八股准备需要分主题、分频率、分复习周期，不只是把资料存起来。

### 3.4 简历与项目包装

用户需要打磨简历，包括：

- 如何讲美团实习
- 是否彻底重写叙述方式，提炼可迁移能力，而不是讲无关技术细节
- cooking 助手是否值得写进简历
- cooking 助手中 LangGraph、主动响应、状态追踪、context awareness 等点如何表达

这些不是传统学习资料，但属于秋招目标的一部分，应能进入计划系统。

### 3.5 当前项目本身：MalDaze / 学习助手

用户已经有一个线条小狗桌宠前端 Maldaze，学习助手会做在这个桌宠里。

相关痛点：

- 过去 spec 写得太简陋，导致一轮生成多个功能后细节质量差
- 前端简陋，需要一个功能一个功能重新设计
- AI 开发后半段需要大量人工试用、记录问题、反馈修改，流程低效
- 用户希望用 OpenSpec + Superpowers + reviewer agent 的流程减少人工审核成本
- 每个闭环要先充分设计、review、修正，再进入实现

这说明学习助手自己的“添加/立项”设计也必须重视需求文档质量、多轮 review 和迭代记录。

## 4. 低优先级但需要留扩展口的主线

除了秋招准备，用户还有三条生活主线，但第一版不应优先实现。

### 4.1 毕业设计

五月开始做，偏写报告和小组作业，压力较轻。可能只需要日程安排和时间分配。

### 4.2 备菜 / 饮食

每三天备一次菜，需要多样、不难吃、不麻烦，并考虑增肌期热量摄入。

普通 chatbot 的痛点是方案不够一次可用，需要大量试错。未来可能让 agent 找有真人反馈的菜谱。

### 4.3 健身

每隔一天去健身房，推日和拉日交替。需要记录器械、重量、次数、动作顺序，并判断下次加重量还是加次数。

健身助手更适合手机端，并需要和桌面端互通。

## 5. 输入类型分类

学习助手的“添加/立项”功能至少需要覆盖这些输入类型：

- 技术方向：例如 agent 开发、后端八股、spec-driven development
- 课程或教程：例如某个 agent 教程、视频课程
- GitHub 仓库：例如 `adongwanai/AgentGuide`、`easyagent`、agent 框架源码
- 文章/网页：例如 agent memory/context 资料、掘金 spec-driven 文章
- 视频/UP 主资料：例如 B 站 up 主“骑猪撞宝马71”的项目视频和包装技巧
- 已有项目：例如 MalDaze、学习助手、cooking 助手
- 简历素材：例如美团实习、LangGraph 项目亮点
- 面试训练项：例如力扣、agent 八股、后端八股
- Obsidian 笔记片段：已有想法、todo、资料索引、项目计划
- 以后可能会看的资源：不应自动进入计划

## 6. 对第一版设计的启发

第一版应优先成为“学习/项目立项与排期入口”，而不是 parser、资料库或今日行动生成器。

当用户添加一个东西时，系统应尽量回答：

- 这是一个要立项排期的目标，还是某个现有计划的资料？
- 如果要立项，目标期限是什么？
- 用户每天/每周可用时间是多少？
- 需要产出什么可验证结果？
- 可以拆成哪些阶段？
- 第一周每天做什么？
- 哪些输入只是资料，不应该变成任务？
- 如果计划太满，哪些可以降级、延后或存档？

计划草案应强调：

- deadline
- 每日可用时间
- 阶段
- 里程碑
- 每日任务
- 缓冲
- 低能量保底任务
- 延期后的重排机制
- 用户确认点

## 7. 明确不应从原文直接推导出的东西

不要把原文中的所有 todo 都变成学习助手需求。

不要把所有资料都变成任务。

不要把“判断值不值得学”和“决定学到什么程度”作为第一版 AI 自动决策能力。

不要把添加功能重新设计成单纯 URL 解析、任务解析、今日行动推荐或资料收集器。

不要把低优先级生活主线提前塞进第一版。

不要让系统比 Obsidian 和已有任务列表更吵、更重、更需要维护。

## 8. 可作为未来优化事项

以下方向重要，但不应阻塞第一版：

- AI 帮用户更深入评估一个学习对象到底值不值得投入
- AI 帮用户判断应该学到什么深度
- 自动读取或同步更大范围的 Obsidian vault
- 自动浏览 GitHub repo 并做深度代码理解
- 自动从小红书、B 站、下厨房等平台收集资料
- 心理 therapist 功能
- 健身手机端
- 备菜/饮食助手
- 多 agent 公司式开发看板
- 自动化产品打磨与 reviewer agent 系统
