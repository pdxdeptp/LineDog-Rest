## Context

The completed v2 slices provide a user-owned study-plan calendar: intake and confirmation, factual Today/Project/Calendar views, deterministic rollover, manual moves, add/delete, rest days, deadline edits, red-state facts, and bounded dialogue preview/apply. The remaining smart-mode slice must be useful without regressing into v1 autonomy.

The v1 Morning Agent is explicitly incompatible with this boundary: it can run weekly review, reschedule tasks, calibrate speed factors, call an LLM, and cache generated briefings. The v2 app currently avoids that path for dashboard refreshes, but `fetchTodayBriefing()` and `/api/today-briefing` still exist. ITEM-004 should add a new v2 smart-mode path rather than repurpose old agents.

## Goals / Non-Goals

**Goals:**

- Persist an off-by-default smart-mode setting.
- Generate a fact-only smart morning briefing from existing v2 views.
- Generate multiple candidate proposal options for morning and after-adjustment triggers.
- Apply only the user-selected proposal after revalidating current plan facts.
- Keep default mode silent and keep ITEM-003 mechanical rules unchanged.
- Verify that v2 smart mode does not call the v1 Morning Agent or old broad chat proposal path.

**Non-Goals:**

- Full-auto smart mode or automatic application of proposals.
- New account, credentials, or external provider setup.
- Retiring/deleting v1 specs or routes.
- Cross-project optimization that cannot be mapped back to an explicit manual operation.
- Replacing the bounded dialogue adjustment path from ITEM-003.

## Decisions

### Decision 1: New v2 smart-mode routes, not `/api/today-briefing`

Add new `study-smart-mode` backend routes and Swift API methods. Do not have the v2 dashboard or smart-mode UI call `/api/today-briefing` because that endpoint is tied to the v1 Morning Agent. This keeps old behavior available for historical specs while isolating v2.

Alternative considered: adapt `run_morning_agent()` to behave differently when smart mode is enabled. Rejected because the old graph's existing contract includes autonomous rescheduling, weekly review catch-up, and speed calibration.

### Decision 2: Smart mode reads v2 facts and produces structured data

The smart-mode service should read from the same v2 fact sources as Today, Project Overview, Calendar, rollover badges, expected-late, and over-capacity. It may run already-defined v2 rollover as part of normal Today fact refresh, but it must not add a new smart-mode mutation before the user presses Apply.

The briefing text should be deterministic or template-based for this slice. If LLM wording is added later, it must be constrained to summarization only and must not choose mutations.

### Decision 3: Proposal options are deterministic preview objects

Represent each proposal option as a structured preview: id/signature, trigger, reason, project id, affected task ids, task/date changes, red-state impact, and a short tradeoff summary. Candidate options should be generated by deterministic heuristics from v2 facts. The implementation can reuse existing preview/apply primitives where they match the proposed action, and add new preview helpers where required.

This makes proposals testable and lets the UI show options side by side. It also prevents the "assistant decided something invisible" problem from v1.

### Decision 4: Apply revalidates the submitted proposal

Apply should accept the selected proposal preview, recompute the equivalent preview from current persisted facts, compare a stable signature, and mutate only if the current signature matches the submitted one. Stale, unsupported, or disabled smart-mode proposals return a no-op response.

This mirrors the safe dialogue apply pattern from ITEM-003 and prevents applying outdated proposals after the user changes the plan.

### Decision 5: Triggering is explicit and gated

Smart-mode proposal generation has two triggers:

- `morning`: enabled smart mode plus lag, expected-late, or over-capacity facts.
- `after_adjustment`: enabled smart mode plus a user/manual adjustment that creates expected-late or over-capacity red state.

Lag/rolled-days are not an after-adjustment trigger. Non-red manual adjustments should not request proposals.

### Decision 6: Swift owns display timing, backend owns eligibility facts

Swift should store smart-mode UI state, display the Settings toggle, fetch morning briefing/proposals only when enabled, and request after-adjustment proposals only after a refreshed v2 fact set indicates the D18 trigger. Backend remains the source of truth for eligibility and returns empty/no-op responses when disabled or not triggered.

## Risks / Trade-offs

- **Risk: Old briefing endpoint is accidentally reused** -> Add source tests and backend tests that smart-mode routes never call `run_morning_agent()`, and ViewModel tests that dashboard/default mode do not call smart endpoints.
- **Risk: Proposal generation gets too broad** -> Keep proposal actions mapped to existing manual operations and record unsupported opportunities as no-op/no proposal.
- **Risk: Applying stale proposals mutates the wrong plan** -> Use stable signatures and revalidation before every apply.
- **Risk: UI becomes noisy** -> Gate after-adjustment proposals strictly on newly created red states and allow ignore/dismiss with no mutation.
- **Risk: Scope expands into full-auto mode** -> Keep full-auto behavior out of this change and leave audit/rollback/default-choice rules to backlog.

## Migration Plan

- Add the smart-mode setting with default `false`; existing users remain in default mode.
- Add new routes and Swift client methods without changing existing v1 routes.
- Add UI surfaces behind the setting.
- Rollback can remove the new routes/UI and leave ITEM-001 through ITEM-003 behavior intact.

## Open Questions
