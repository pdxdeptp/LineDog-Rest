## Context

The desk-pet Dashboard left sidebar is labeled "计划" and renders incomplete EventKit reminders from the selected reminder list. Production fetching currently combines overdue reminders, today's reminders, and reminders due from today through seven days ahead. The UI copy also says "七日内", so reminders scheduled beyond the next week are intentionally invisible in the sidebar.

This change keeps the existing EventKit-backed model and local grouping behavior, but extends the forward-looking reminder horizon to the next three months.

## Goals / Non-Goals

**Goals:**
- Show incomplete selected-list reminders due within the next three months in the desk-pet Dashboard "计划" sidebar.
- Continue showing overdue incomplete reminders and today's `#日常` reminders.
- Keep grouping by local calendar day and sorting by due date.
- Update user-facing sidebar copy and tests to reflect the longer horizon.

**Non-Goals:**
- Do not add pagination, filtering controls, search, or a user-configurable horizon.
- Do not change EventKit permission handling, selected reminder list persistence, editing, delete, complete, or postpone behavior.
- Do not change smart reminder creation or natural-language parsing.

## Decisions

1. Use a fixed three-month horizon in the existing EventKit query.
   - Rationale: the user asked for "未来三个月" as the new baseline, and a fixed horizon keeps the sidebar behavior simple.
   - Alternative considered: add a setting for seven days / one month / three months. Rejected for this change because it expands UI scope and preference persistence without being requested.

2. Represent the horizon as a named constant near the EventKit fetch code.
   - Rationale: this makes the three-month window discoverable and avoids another magic number in the fetch predicate.
   - Alternative considered: inline `.month, value: 3`. Rejected because tests and future maintenance benefit from a named policy.

3. Keep the existing merge input shape while renaming only where it clarifies intent.
   - Rationale: `DeskReminderSidebarMerger` already handles dedupe and sorting independent of the horizon length.
   - Alternative considered: introduce a new planner-window type. Rejected because the behavior remains a simple in-memory merge.

## Risks / Trade-offs

- [Risk] Fetching three months of reminders can return more rows than the current week, making the sidebar longer. -> Mitigation: the sidebar is already scrollable and still scoped to one selected reminder list.
- [Risk] A longer window may include more routine recurring reminders generated by EventKit. -> Mitigation: preserve the existing incomplete-reminder predicate and selected-list scope; do not expand to all lists.
- [Risk] Tests that encoded "week" naming could become misleading. -> Mitigation: update tests and comments to describe the upcoming-window behavior rather than a seven-day window.

## Migration Plan

No data migration is needed. Rolling back means restoring the previous seven-day EventKit query horizon and sidebar copy.
