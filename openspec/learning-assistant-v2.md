## 1. 背景:为什么要 v2

学习助手 v1 围绕**三个自治 LLM agent**(`daily-morning-agent` 晨报、`conversational-planner` 对话、`material-ingestion` 摄取)加 `weekly-review-agent` 周报构建。这些 agent 主动运行、凭空生出"自己的能力",底下没有一个用户能手动操作的模型托底,系统行为脱离用户预期。用户判断 v1「偏离太远」,要求从头重新梳理用户故事,先想清楚要什么,再决定 PRD,再回去改代码。

v2 把学习助手重新定位为:

> **一个「带 LLM 排课能力的学习计划日历」,不是一个「会主动管你的 AI 伙伴」。**

---

## 2. 定盘星:LLM 的边界

这是整个 v2 的核心设计纪律,防止再次偏离:

```
LLM 只做两件事:
  1. URL → 拆解生成初始计划
  2. 对话式调整计划(自然语言指令)

LLM / 助手不做(默认模式):
  ✗ 主动生成晨报
  ✗ 主动重排、主动提醒、主动说话
  ✗ 自动改你的计划
```

调整永远由用户发起,系统只负责级联与如实显示。

**智能模式同样受这条纪律约束:** 智能模式不是另一个系统,它只是「助手替你做你本来能手动做的事」。智能模式能做的每一件事,都必须能对应到默认模式里一个具体的手动操作——只要守住这条,它就不可能长出 v1 那种失控的 agent 行为。v1 偏离,正是因为 agent 凭空长出了能力、没有手动模型托底。

## 3. 两个模式

```
┌─────────────────────────────────────────────┐
│  默认模式  = 手动为主                          │
│             LLM 只做 URL 拆解 + 对话调整       │
│             系统只如实显示,从不自作主张        │
├─────────────────────────────────────────────┤
│  智能模式  = 默认模式 + 一个「提议引擎」        │
│             设置里 opt-in                     │
└─────────────────────────────────────────────┘
```

## 4. 用户故事 US-1 ~ US-21

### 一次性设置
- **US-1** 我设定每日学习时长上限,系统据此决定每天能排多少任务。(全局,固定项,不是开关)
- **US-20** 我可以设置每周固定放空日(默认周六,可改),也可以手动把某天标为放空日;每多一天放空,后续未完成任务自动 +1 天后推(语义 = 每周少 1 天)。

### 添加项目
- **US-2** 我贴入一个 URL + 截止日期,LLM 分析内容,自动拆成有序任务,每个任务带时长估算,并铺到日历上。**(parse 走 D29 decomposition pipeline;parse 前可能先有 2~3 个 guided 问题——见 OQ3,UX 细节探索中)**
- **US-3** 计划生成后进入「审阅」状态,我用四种方式精调:时间线拖拽 / 每日列表编辑 / 对话 / 日历块。
- **US-4** 我能改任何任务的时长估算(LLM 初估值不一定准)。
- **US-5** 我确认后,计划才正式生效进入日常。

### 日常使用
- **US-6** 每天打开,默认看「今日视图」——今天所有任务(跨项目)的清单。
- **US-7** 我勾选完成任务,进度随之更新。
- **US-8** 今天没勾完的任务,第二天自动滚入次日列表,无需我操作(不级联)。
- **US-21** 一个任务被自动滚入 ≥ 3 天时,UI 上自动出现「已滚 N 天」小标;智能模式打开时,晨报也会专门点名提它。

### 随时调整
- **US-9** 我能拖动任意任务改日期;同项目后续任务整体跟着平移相同天数(级联,仅限本项目)。
- **US-10** 我能改任意项目的截止日期。
- **US-11** 红色是**状态判定**:任何任务落在项目截止日之后 → 项目标红「预计超期」;某天任务总时长超过日上限 → 那天标红超额。不论原因(初始排布 / 滚入 / 拖拽级联 / 增删),系统只如实显示。
- **US-15** 我能手动增 / 删单个任务,不只是移动。
- **US-16**「对话」调整入口是一个对话框,我用自然语言说「把英语项目整体推迟一周」,助手执行(走路线甲——见 D25,先给预览 + [应用])。

### 全局感知
- **US-12** 我能切到「项目总览」看每个项目的进度条和截止日期。
- **US-13** 我能切到「日历视图」看未来几周的任务分布和负荷。
- **US-14** 项目所有任务勾完 → 自动归档,保留完成记录(可回顾)。

### 智能模式(opt-in)
- **US-17** 我能在设置里打开「智能模式」;关闭即默认模式,打开后多出一个晨报。
- **US-18** 晨报汇总今日视图 + 落后 / 预计超期 / 超额情况——只是把默认模式里已有的信息总结呈现,不产生新数据。
- **US-19** 助手在两个时机给出多个候选调整方案 + [应用]:**晨报里**(触发点①,扫到落后 / 预计超期 / 超额时)、和**手动调整制造红色之后**(触发点②,仅红色触发——见 D18);每个方案带影响预览,我选其一或全部忽略,修改永远由我触发。

## 5. 已定的关键决策

| # | 决策 | 理由 |
|---|------|------|
| D1 | 需求是**完成驱动**("我做完了"驱动),不是掉课驱动 / 外部变化驱动 | 用户明确选择 |
| D2 | 计划粒度 = 排出**每天做什么任务**;任务时长 LLM 初估、用户可改 | 时长是日历排布的一等公民 |
| D3 | 约束维度 = **每日学习时长上限**,不限同时进行的项目数 | 用户按时长而非项目数管理负荷 |
| D4 | 项目内任务**有顺序依赖**,项目间**无依赖** | 决定级联逻辑的边界 |
| D5 | **截止日期添加项目时给(必填)、可编辑** | v2 核心定位是"带 LLM 排课的日历",日历隐喻天然需要时间锚——D24 排布、红色判定、"落后"度量全靠截止日定义。"无截止日"型项目(随便学)是 backlog 项(见 §9),走另一套 forward-pack 排布 |
| D6 | 四种调整交互(拖拽 / 列表 / 对话 / 日历块)**全部支持** | 用户要 ABCD 都做 |
| D7 | **默认模式下**,调整时助手**沉默不说话**,但所有连带变化要跟着变 | 智能模式下,触发点② 按 D18/D19 接手给提议;D7 只管默认模式行为 |
| D8 | 今日入口是**今日视图**;项目总览、日历视图也都要 | 今日视图为主入口 |
| D9 | 没勾完的任务**自动滚入次日,且不级联** | 滚入只往后挪一点,顶多与次日同项目任务叠着,先做老的即可,不会乱序 |
| D10 | **用户主动改任务日期**(不论从哪种交互——时间线拖拽 / 日历块拖动 / 列表改日期 / 对话指令)**触发级联,限定在本项目内** | 改日期带级联 → 项目内顺序天然保持;级联规则跟着"用户改了日期"这个动作走,与 UI 表面无关 |
| D11 | 改日期级联用**方式甲**:后续任务整体平移相同天数(平移量 = 用户施加的日期变化 Δ) | 所见即所得,不用脑补哪个会撞;位移量来自用户动作,系统不"算" |
| D12 | 「顺序错乱」这个危险 corner case,被 D9+D10 这对规则**彻底消除**(增删情形见 D23) | 拖拽保序、滚入只堆叠不乱序;增删任务的顺序错乱后来由 D23「顺序由日期推导」自然消解 |
| D13 | **系统只执行用户动作的字面意义 + 已定机械规则(滚入 D9、拖拽级联 D10/D11);不额外搬动任务**。红色状态(超期 / 超额)如实显示,由用户(默认模式)或智能模式提议引擎(智能模式)处理 | 「合法但落后」的状态,系统不替用户解「怎么塞回去」;"不额外搬动" 同时管两件事——无人触发时不自主重排 + 有人触发(增删)时也只做字面动作不偷偷补刀;D9/D10/D11 是已定明文机械,不算"额外" |
| D14 | 两个模式:默认模式 / 智能模式(设置 opt-in) | — |
| D15 | 智能模式走**路线甲**(提议 + 用户确认),不走路线乙(直接改) | 计划的最终修改权永远在用户手里;按下应用那一下永远是用户。乙把最后一道闸交给 LLM,等于给失控留后门 |
| D16 | 助手提议有多个选项时,**几个方案全部并列给出**,每个带 [应用] | 用户明确要求 |
| D17 | spec 组织采用**策略 B**:建全新 v2 spec id;v1 的 learning spec 在 v2 全部实现后整体退役 | v2 是架构级重做,不该套在 v1 旧骨架上。**退役清单(9 个):** `assistant-panel-ui` / `conversational-planner` / `daily-morning-agent` / `ingestion-progress-sse` / `learning-data-layer` / `learning-preferences` / `material-ingestion` / `progress-feedback` / `weekly-review-agent`(其余 5 个 spec 是桌宠 / 喝水提醒,不动)。**退役方式:** 文件和内容全保留(历史可追溯) + 顶部加废弃说明,指向 v2 替代 spec(`study-plan` / `study-views` / `study-plan-adjustment` / `study-smart-mode`)——读者一眼识别为非 active,但仍可查阅 |
| D18 | 智能重排(提议引擎触发点②)在手动调整之后触发,**且仅当该调整制造了红色问题(超期 / 超额)时**才触发;无红色问题则助手不提议 | 没撑爆任何一天还跳出来建议「我觉得这样更好」= 唠叨;助手只在「确实出问题」时出手 |
| D19 | 智能模式下机械级联(D10/D11)**行为不变**——照常瞬时、确定性、限本项目;智能重排只**叠加在级联之后**作为提议,不改变拖动本身的即时行为 | 自动那层永远在本项目内、永远确定;跨项目、有判断的那层永远是提议。两层分开,才不会重蹈 v1「两层混在一起、都让 agent 跑」的覆辙 |
| D20 | **删任务**:只删该任务,后续任务不动(无级联);腾出的时长只是让那天变轻 | 删除只减负、永不制造红色,原地不动永远是合法状态;回拉后续 = 系统主动重排,违反 D13;删除也没有拖拽那样的手势位移量可回放 |
| D21 | **插任务落点**:用户填「标题 + 时长 + 落点日期」,沿用拖拽手势把它丢到某天;项目内顺序由日期推导,不设单独的「选顺序槽」UI | 贴合 v2「日历是真相、任务钉在日期上」的既有心智模型;顺序由日期推导,让「顺序错乱」结构上不可能发生 |
| D22 | **插任务级联**:不级联,后续任务不动;新任务堆叠到目标日,撑爆 cap 那天标红、落在截止日后项目标红——只如实显示 | 插入没有手势位移量,任何「让位」级联都要系统自己算 Δ = 重新打包,违反 D13;与 D20 镜像对称 |
| D23 | 增删不引入新的「顺序错乱」corner case:删除不重排、插入顺序由日期推导 → 「顺序 = 时间序」不变量构造性保持,D12 的保护自动覆盖 US-15 | corner case 在系统层面根本不存在,无需额外机制 |
| D24 | **初始排布算法**:任务在 [今天, 项目截止日] 区间内的**非放空日**(见 D26)上平均摊开(任务时长不等,匀分是近似);多项目叠加用 α —— 每个项目摊自己的,不看其他项目的已有负荷,合并后超上限标红 | 规则一行话能说清,预测性好;如实显示而非系统替用户回避红色;删项目时不抖动(α 下删英语不会让数学自动重摊) |
| D25 | **对话(US-16)统一走路线甲**:每条指令先给预览「我将这样改:…」+ [应用] 按钮,用户点了才真改 | 对话是 LLM 替用户做事,与智能模式同源——D15「按下应用那一下永远是用户」对对话同样成立;简单指令的预览也就一行字,不显烦 |
| D26 | **放空日 = 那天日上限 = 0 的天**,系统不在放空日排任务。默认每周六放空,用户可在设置里改(增 / 减 / 换 weekday) | 用户需要"不学习的日子"的合法表达;以"日为颗粒度"的开关,简单,跟日上限同维度 |
| D27 | **新增放空日 → 全局 +1 天 cascade**:任何一天新成为放空日(weekly 设置增 weekday → 该 weekday 的每次未来发生;手动加单天 → 选的那一天)→ 从该天起做一次 +1 天 cascade(跨所有项目所有未完成任务)。多个新放空日按 chronological 顺序逐一应用,效果累积——加周六放空 N 周项目 = 整体推 N 天。**取消**放空日不 cascade,只是从此允许排到该天 | 用户给了"日粒度位移"的明确动作(类比 D11 的 Δ),系统照办;累积位移让冲突自动消化,语义 = "每周少 1 天" |
| D28 | **滚动任务可视化(阈值 3 天)**:任务滚动 ≥ 3 天 → UI 显示「已滚 N 天」小标(默认模式);智能模式晨报对滚动 ≥ 3 天的任务点名提议处理(触发点①信号,与「落后」并列)。**滚动天数** = 任务被 D9 自动滚入的累积次数;用户主动的日期变化(拖拽 / 对话 / 放空日 D27 cascade)重置 baseline,勾完即归零并移除 | 给用户两层非强制信号:不看晨报也能在任务上看到、看得到的人能更早介入。3 天 = "自觉一次"的合理缓冲。系统不动 D9 本身、不替用户决定,只是把信号亮足 |
| D29 | **URL → 计划走多步 decomposition pipeline**:LLM 不一把出全部计划,拆成 5 步——(1) 抽结构 → (2) 估难度 → (3) 估时长 → (4) 合并成有序任务 → (5) 铺日历(D24 处理)。资料类型(Coursera / PDF / B 站 / 文档站)第一版只做 1~2 种主流类型的特化 handler,其余走通用 fallback。**parse 前的 guided 问题(B)单独走 OQ3** | 一把跑容易"短上下文 + 全责"导致质量差;拆步后每步任务窄、上下文聚焦、可独立调试,质量上限高。资料类型特化(E)工程上有限投入,主流场景覆盖即可 |

## 6. 智能模式设计

智能模式在默认模式之上,只多了**一个提议引擎**。「晨报」「重排」「自动调整」不是三个独立功能,而是同一个引擎被不同事件唤醒:

```
              ┌──────────────────────────────────────┐
              │   提议引擎                            │
              │   检测问题 → 算出多个候选方案 → 等你确认  │
              └──────────────────────────────────────┘
                   ▲                          ▲
        触发点 ①   │                          │   触发点 ②
        每天打开 ──┘                          └── 手动调整一个项目之后
        = 晨报                                    = 智能重排
        (全局扫一遍)                              (扫这次调整的连锁后果)
```

提议示例(检测到「英语项目落后 2 天」):

```
┌────────────────────────────────────────────┐
│ 方案 A  后移这 3 个任务 → 截止日 6/2   [应用] │
│ 方案 B  延截止日 3 天 → 6/5            [应用] │
│ 方案 C  本周每天 +30 分钟              [应用] │
└────────────────────────────────────────────┘
点哪行的 [应用] / 全部忽略 —— 按下去那一下永远是用户
```

**触发条件(D18 / D19):** 触发点② 的智能重排只在手动调整**制造了红色问题(超期 / 超额)时**触发;调整没撑爆任何一天、也没超期 → 助手不提议。机械级联(D10/D11)在智能模式下**照常瞬时发生**,智能重排只叠在它之后作为提议——拖动本身的即时行为不变。

**「滚动天数 / 落后」定义:** 一个任务被 D9 自动滚入的累积次数 = 该任务的**滚动天数**(用户主动的日期变化——拖拽 / 对话 / 改截止日 / 放空日 D27 cascade ——重置 baseline,勾完即归零并移除)。项目的「落后 N 天」= 项目内所有任务滚动天数的最大值。本质是"没做完任务"造成的延期;用户主动延期不算。落后**只在晨报(触发点①)里**作为助手提议的触发信号出现;它**不**触发触发点② —— D18 已限定触发点② 只在红色(超期 / 超额)时点火。任务级滚动信号见 D28。

智能模式相关用户故事见 §4 的「智能模式(opt-in)」子节(US-17~19)。

## 7. Open Questions

### OQ3 — Parse 质量 · guided clarification 的 UX 设计 · 🟡 探索中(2026-05-23 开)

**背景:** v1 大痛点是 URL → 任务计划的 one-shot 质量差,只能靠多轮对话救场。v2 提了 5 个改进方向,处置如下:

| 方向 | 状态 | 落地 |
|------|------|------|
| **A** · decomposition pipeline(生成本身拆多步) | ✅ 已定 | **D29** |
| **B** · guided clarification(parse 前问 2~3 个 targeted 问题) | 🟡 探索中 | **本 OQ3**,闭合后落 D30 |
| **C** · self-critique loop | ❌ 不上 | B 充分后边际收益小 |
| **D** · 用户校准 prior(长期数据) | ⏸ backlog | 无历史数据,做不了;入 backlog 留作未来 |
| **E** · 资料类型特化 handler | ✅ 部分定 | 第一版只做 1~2 种主流(见 D29 备注) |

**B 待定的 5 个子问题:**

### OQ1 — US-15 的尾巴:增删任务对项目内顺序与排期的影响 · ✅ 已闭合(2026-05-22)

**问题:** 删任务、在项目中间插任务,会动到项目内的顺序和后续排期,需要一套像 D10/D11 那样的规则。

**闭合思路 —— 答案不靠发明,是两条既有纪律的直接推论:**

- **纪律一(D13 + 定盘星):** 系统只执行用户动作的字面意义 + 已定机械规则;不额外搬动任务。
- **纪律二(D11 的本质):** 改日期级联之所以成立,是因为位移量来自用户的明确动作——用户(拖动 / 列表改 / 对话)指定了一个 Δ 天数,后续就平移 Δ 天。系统从不「算」该移多少,只是回放用户给的 Δ。

> **关键洞察:级联需要一个「用户给定的位移量」。删除和插入都没有这个手势量。**
> 任务在 v2 里是钉在日期上的对象;删一个 2h 的任务 ≠ 空出「一整天」,插一个任务也不是「整体移 N 天」。
> 想让后续任务为删/插「让位 / 补位」,系统就必须自己算一个位移 = D13 明令禁止的「重新打包」。
> → 这直接堵死「增删带级联」,定下 D20 / D22 全部「就地做、不级联」。

**OQ1a — 删任务 → D20:**

```
删「任务3」(2h)   前: 日5[T1][T2]  日6[T3]  日7[T4][T5]  日8[T6]
✗ 回拉级联:T4~T6 前移补空 → T3 占 2h≠整天,要重新打包 = 违反 D13
✓ 原地不动: 日5[T1][T2]  日6(空,变轻)  日7[T4][T5]  日8[T6]
```

删除只减负、永不制造红色 → 原地不动永远合法,系统没有「需要去修」的压力。想压缩计划,用户自己拖拽 / 对话。

**OQ1b — 插任务落点 → D21:** 用「日期推导顺序」而非「显式选槽」。用户把新任务丢到某天(沿用拖拽手势),项目内顺序 = 它在日期序里的位置,自动得出。依赖顺序是 LLM 生成时的概念;计划确认后用户手编,操作模型就是「日历上的任务」——这正是 v2 其余部分已选定的心智模型。插入需用户填「标题 + 时长 + 落点日期」(时长无 LLM 初估,直接给)。

**OQ1c — 插任务级联 → D22:**

```
插「复习」(1.5h, 落点日7)   前: 日6[T3]  日7[T4 1h][T5 1h]  日8[T6]
✗ 插入级联:T4 之后整体后移让位 → 移多少要系统算 = 重新打包 = 违反 D13
✓ 堆叠不动: 日7[T4][T5][复习] = 3.5h > cap → 日7 标红;日8[T6] 不动
```

与删除镜像对称:操作只做字面的事,红色状态如实显示。想要 Gantt 式「插入推后继」,那是拖拽或智能模式提议引擎的活。

**OQ1 corner case → D23:** D12 宣称 D9+D10 消除了「顺序错乱」,但没覆盖「插入」。上面的推荐让它自动补上:删除不重排 → 不可能乱序;插入顺序由日期推导 → 「任务排到依赖项之前」在系统层面根本不存在(顺序就是日期定义的)。最多是用户把任务丢错天——用户的判断失误,系统如实显示,符合定盘星。

**与智能模式的衔接 —— D18/D19 免费覆盖,无需新规则:**

```
删任务 → 只减负、永不制造红色 → D18 不触发 → 助手静默
插任务 → 可能制造红色:堆叠撑爆 cap(超额红)/ 新任务丢在截止日后(预计超期红)
       → 命中红色 → D18 触发,提议引擎给多个重新分布方案
```

机械层(增删本身)永远瞬时、确定、原地;判断层(要不要重排)永远是提议——两层依旧干净分开,正是 D19 的精神。OQ1 决策能严丝合缝插进 D18/D19 而不用改它们,本身就是 v2 设计自洽的证据。

**边界情形(小,不阻塞,留给对应 change 的 propose 阶段细化):**
- 插入落点 ≥ 今天,不能排到过去。
- 删光一个项目的全部任务 → 项目变空;「删空项目本身」超出 US-15「增删单个任务」范围。
- 删掉最后一个未完成任务:归档判定建议统一为「剩余未完成任务 = 0 → 归档」,无论最后一个是勾完还是删掉(US-14 微调,propose `study-views` 时定)。
- 进度 = 已完成数 / 当前总数,增删后自动重算,无需专门规则。
- 已归档项目只读,不支持增删。

## 8. 实现路线:切成一系列小 change

v2 是 epic,不是一个 change。按依赖关系切片(对应 4 个 v2 spec id):

```
study-plan  (US-1~5)              ← 地基,无依赖
  计划怎么来:每日时长上限·URL拆解·审阅·改时长·确认生效
        │
        ├──────────────────────┬───────────────────────┐
        ▼                      ▼
study-views (US-6,7,12,13,14)   study-plan-adjustment (US-8~11,15,16)
  今日视图·勾选·项目总览·         滚入·拖动级联·改截止日·
  日历·项目归档                  超期标红·增删任务·对话框
  (依赖 study-plan)              (依赖 study-plan)
                                       │
                                       ▼
                                study-smart-mode (US-17~19)
                                  设置开关·晨报·多方案提议
                                  (依赖 study-plan-adjustment)
```

- `study-views` 与 `study-plan-adjustment` 在接口层互不依赖,可并行开发;但 adjustment 的可视效果(滚入列表、红色标记)落在 views 的 UI 上,完整测试需要 views 就位。
- **调整机制归属:** 拖拽 + 级联(D10/D11)和增删(D20–D22)被 `study-plan` 的审阅期(US-3「四种方式精调」)和 `study-plan-adjustment` 的日常(US-9、US-15)同时需要。**机制本身归 `study-plan` 实现**(审阅期就要用上),`study-plan-adjustment` 复用机制 + 加上日常状态管理(滚入、持久红色、改截止日)。具体边界 propose `study-plan` 时再确认。
- `study-plan-adjustment` 装了 6 个故事,实现时可能还要再切(如对话框 US-16、增删 US-15 各自独立)——propose 它时再定。US-15 的增删规则已由 D20–D23 定死,不再有设计阻塞。
- 每个 change 归档时,其 `specs/<spec-id>/` 并入 `openspec/specs/<spec-id>/`,逐步砌起 v2 的 4 个 spec。
- 第一个 change 很可能是 `study-plan`(US-1~5,地基无依赖,已可独立 propose);OQ1 / OQ2 均已闭合,后续 change 不再有未决设计问题阻塞。

## 9. 关联 backlog

- **全自动智能模式 + 全自动对话** —— 智能模式从路线甲(每方案 [应用])进化到自动应用、事后通知;对话(US-16)从路线甲(预览 + [应用])进化到直接执行。两条线共用一个安全底座:
  - **rollback** —— 用户感觉不对,一键回退到上一稳定状态
  - **audit trail** —— 谁(智能模式 / 对话 / 用户)/ 何时 / 改了什么的完整记录
  - **自进化** —— 从 rollback 信号反向训练判断:被回退的就是错的,系统学着不再那么选
- **前置门槛:** rollback 机制必须先存在,否则自动化不解锁(否则就是 v1 失控的翻版)。
- **次级前置:** 智能模式的多方案并列(US-19)在全自动下要先有「默认选哪个方案」的决策规则。
- 详见 `openspec/backlog.md` 的 `study-smart-mode` 模块。

- **无截止日项目支持** —— 允许"随便学"型项目(无明确截止日),走 forward-pack 排布(从今天起按日上限往后塞,做完为止)。需要为这类项目重定义红色判定("预计超期"消失,只剩"超额")和"落后"度量(可能不定义,只显示进度)。详见 `openspec/backlog.md` 的 `study-plan` 模块。
